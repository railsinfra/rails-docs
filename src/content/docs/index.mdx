    <li><strong>Who uses it:</strong> product and engineering teams building fintech, internal tools, or regulated money flows.</li>
    <li><strong>Why it exists:</strong> keep identity, transfers, and audit history explicit instead of hiding them inside one application database.</li>

  </ul>

<div class="rails-intro__divider" />

<p class="rails-intro__body">
  <strong>Rails</strong> is banking infrastructure: identity, accounts, and a double-entry ledger behind one
  gateway. Use it when you need code-driven control of users, balances, and money movement that must
  complete fully or not at all—without packing all of that into a single database or deployable.
</p>

## What you will learn

      <h3>How the platform is shaped</h3>
    </div>
    <p>
      Users, accounts, and ledger run as separate services so a failure in one area is less likely to take
      down everything, and each service has a clear contract. Guides explain how the pieces connect and
      how to run samples or the open-source stack on your laptop.
    </p>
  </article>
  <article class="rails-intro__learn-card">
    <div class="rails-intro__learn-head">

      <h3>How to integrate from code</h3>
    </div>
    <p>
      Official SDKs are built from the same API description as this site, so method names and routes stay
      aligned with what you read here. Use guides for concepts, then open the API Reference tab for paths,
      request bodies, and copy-ready examples.
    </p>
  </article>
